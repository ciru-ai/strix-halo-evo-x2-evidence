Evidence Story

NPU-aware daily-driver evidence leads this package, followed by optimized Strix Halo model results from Chadrock/Qwopus, CrownV7, and Gemma QAT/MTP.

  • NPU sidecar lane: run cron jobs, Hermes tasks, and benchmark supervision on an always-on NPU service with very low measured overhead on the main CPU/iGPU path.
  • IOMMU-on rationale: keep the NPU available as a separate always-on execution lane without slowing normal CPU/iGPU work when side tasks run.
  • 27B Chadrock/Qwopus: same HumanEval workload, higher score, and about 2.85x lower recorded request-generation time than the stored original Qwopus3.6 27B v2 Q5_K_M run.
  • 35B Chadrock/Ace: strong HumanEval quality paired with over-90 and over-100 tok/s quality-run speed evidence.
  • CrownV7: optimized Qwen3.6 35B Strix Halo route with strong 128k prefill/decode evidence and useful tool/agent benchmark anchors.
  • Gemma 4 26B: new QAT/MTP API speed row implies about 122.8 decode tok/s after TTFP.
  • Gemma 4 12B QAT/MTP: the n-max 5 speed configuration also scored 0.9329 HumanEval+ on a full 164-task EvalPlus run.

System Snapshot To Include

Hardware / Firmware

  • GMKtec NucBox_EVO-X2, SKU EVO-X2-001, hardware version 1.0
  • AMD Ryzen AI MAX+ 395 with Radeon 8060S
  • 16 cores / 32 threads
  • 123 GiB visible system memory, 128 GB class
  • BIOS VRAM setting: 2 GB; the workload relies on UMA access to the full 128 GB class memory pool rather than reserving a large fixed VRAM carveout
  • Firmware EVO-X2 1.12, firmware date 2025-12-09

Software / Acceleration

  • NixOS 26.05 pre-release build 26.05pre987561.1c3fe55ad329
  • Linux 7.0.1
  • RADV STRIX_HALO, Mesa 26.0.5, Vulkan 1.4.341 instance
  • AMD Strix/Krackan/Strix Halo NPU exposed through /dev/accel/accel0
  • IOMMU enabled; kernel cmdline includes iommu.passthrough=0
  • IOMMU-on rationale: keep an always-on NPU sidecar available for cron, Hermes, and benchmark supervision work without materially slowing the CPU/iGPU path
